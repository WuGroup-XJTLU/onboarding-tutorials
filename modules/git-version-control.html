Version Control with Git

Version control is the memory of a project: every change recorded, every past state recoverable, and a clear answer to "which version of the code made this figure?". Git is the tool the group — and almost all of science and software — uses for it. Later modules assume you work inside version control; this is where you learn to.

Learning objectives

  • Explain what version control is and why every project needs it
  • Configure Git and create or clone a repository
  • Use the core cycle: status → add → commit, and read the history
  • Keep data, outputs and secrets out of the repo with .gitignore
  • Work on branches and merge them back
  • Collaborate on GitHub with remotes and pull requests
  • Recover from mistakes — unstage, discard, amend and revert

Prerequisites

  • Module 01 — Unix/Linux Basics (the shell, paths, editing files)
  • Module 02 — Bash Scripting (helpful — the scripts you write are what you'll track)

Estimated time

  • 12–15 hours

Overview

Without version control, a project drifts into a folder full of analysis_final.py, analysis_final_v2.py, and analysis_really_final.py — and no reliable way to say what changed between them or to get a working version back after a bad edit. Version control solves this by recording your project as a series of snapshots (commits), each with a message, an author, and a timestamp. You can inspect any past state, compare any two, and undo with confidence.

For research this is not a nicety — it is part of doing reproducible work. When a reviewer asks how you produced a result two years from now, the commit that generated it, and the exact code at that point, are still there. Git is the dominant version-control system; GitHub is where the group hosts its repositories and reviews changes. This module teaches the everyday Git you need before the modules that assume it — Module 07 in particular tells you to "work inside version control" so every AI-assisted change is a reviewable diff.

1 · Set up and create a repository

Git usually ships with Linux and macOS (check with git --version; install from git-scm.com otherwise). Tell Git who you are once, so your commits are attributed correctly:

git config --global user.name  "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main   # name the first branch "main"

You then either start a repository in an existing folder, or copy an existing one from GitHub:

git init                 # turn the current directory into a repository
# ...or...
git clone https://github.com/WuGroup-XJTLU/<repo>.git   # copy an existing one

2 · The core cycle: add, commit, push

Git has three places a change can live: your working tree (the files as they are on disk), the staging area (changes you've marked to include in the next snapshot), and the repository (the committed history). git add moves changes into staging; git commit records staged changes as a permanent snapshot; git push sends your commits to the shared copy on GitHub.

In practice the loop is short and you run it constantly:

git status                 # what's changed, what's staged
git add analysis.py        # stage a specific file (or `git add -p` to pick hunks)
git commit -m "Add RDF calculation for the solvent box"
git log --oneline          # a compact history
git diff                   # unstaged changes;  git diff --staged for staged ones

Write commit messages for the next reader

A good message says why, not just what: "Fix off-by-one in bin edges (RDF was shifted by half a bin)" is worth ten "update"s. Commit in small, logical steps — one idea per commit — so the history stays reviewable and any single change is easy to undo.

3 · What not to commit: .gitignore

A repository is for source — code, scripts, small input files, and text you author. It is not for large data, generated outputs, or secrets. Committing a multi-gigabyte trajectory or a directory of figures bloats the repo for everyone and can't be fully undone. List the things Git should ignore in a .gitignore file at the repo root:

# .gitignore — patterns Git will not track
*.dcd            # large MD trajectories
*.xtc
*.log
__pycache__/     # generated Python bytecode
results/         # generated figures and data
*.key            # never commit secrets or credentials
.env

Data and secrets stay out of Git

Keep raw and large data in your allocated storage (on the cluster, see Module 06 — HPC & SLURM Job Submission), not in the repo. Never commit passwords, API keys, or tokens — if you do, treat them as compromised and rotate them. Commit the script that generates a figure, not a hundred copies of the figure itself.

4 · Branching and merging

A branch is an independent line of work. Instead of editing main directly, you make a branch for a feature or experiment, commit freely, and merge it back once it works — so main always stays in a known-good state.

git switch -c feature/rdf-analysis   # create and switch to a new branch
# ... edit, add, commit as usual ...
git switch main                      # go back to the main line
git merge feature/rdf-analysis       # bring the finished work in

If the same lines changed on both branches, Git reports a merge conflict and marks the spots with <<<<<<< / ======= / >>>>>>>. Open the file, keep the correct combination, delete the markers, then git add the file and git commit to finish the merge. Conflicts are normal and local — nothing is lost.

5 · Collaborating on GitHub

GitHub stores a shared copy of a repository (a remote, conventionally named origin) and adds review tools on top of Git. The collaboration loop:

  • Clone the repo (or fork it, if you don't have write access), then make a branch for your change.
  • Push your branch: git push -u origin feature/rdf-analysis.
  • Open a pull request (PR) — a proposal to merge your branch, showing the full diff. Collaborators review it, request changes, and approve.
  • Once approved, the PR is merged into main; everyone else git pulls to catch up.

Pull requests are how the group reviews code before it lands — the same reviewing discipline Module 07 applies to AI-generated diffs. The group's repositories live under the WuGroup-XJTLU organisation.

6 · Undoing mistakes

Almost anything in Git is recoverable. The common fixes:

  • Unstage a file (keep the edit, just don't commit it yet): git restore --staged <file>.
  • Discard uncommitted changes to a file (irreversible for that edit): git restore <file>.
  • Fix the last commit (wrong message, forgot a file): stage the fix, then git commit --amend.
  • Undo a commit that's already shared: git revert <hash> records a new commit that reverses it — safe for history others have.

Don't rewrite shared history

Amending or rebasing commits you've already pushed rewrites history other people may have based work on. Use git revert (a new, forward commit) for anything already on the shared remote; save --amend and rebase for commits that are still only local.

Resources

  • Pro Git — Scott Chacon & Ben Straub. The definitive book, free online; chapters 1–3 cover everything here. git-scm.com/book
  • GitHub — Git & GitHub docs — practical guides for clone, branch, push, and pull requests. docs.github.com/en/get-started
  • Git official documentation — the reference for every command and option. git-scm.com/doc
  • Oh Shit, Git!?! — plain-language recipes for getting out of common Git tangles. ohshitgit.com
  • Software Carpentry — Version Control with Git — a gentle, hands-on lesson for researchers. swcarpentry.github.io/git-novice

Exercise / Deliverable

  1. Create a repository (git init) for a small project — e.g. the summarise.sh script from Module 02 — and make your first commit.
  2. Add a .gitignore that excludes a generated-output directory and a data file type, and confirm with git status that those files are no longer tracked.
  3. Make two or three further commits with clear messages, then show the history with git log --oneline.
  4. Create a branch, make a change on it, switch back to main, and merge the branch in.
  5. Create an empty repository on GitHub, add it as a remote, git push your work, and open a pull request for a small follow-up change on a branch.

Deliverable: the GitHub repository URL, the output of git log --oneline showing your commits and the merge, your .gitignore, and the link to your pull request.
