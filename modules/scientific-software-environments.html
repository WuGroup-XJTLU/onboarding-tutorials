Scientific Software & Environments

"It works on my machine" is not a result anyone can reproduce. This module is about installing scientific software into isolated, reproducible environments — so the exact software behind a calculation is recorded, shareable, and the same on your laptop as on a colleague's.

Learning objectives

  • Explain why isolated environments matter for reproducibility
  • Create and manage environments with conda / mamba
  • Pin and share an environment with environment.yml
  • Choose sensibly between conda and pip
  • Install the group's scientific stack
  • Understand how local environments relate to the cluster

Prerequisites

  • Module 04 — Python for Scientific Computing (you'll manage its packages)
  • Module 01 — Unix/Linux Basics (the shell)

Estimated time

  • 8–12 hours

Overview

Different projects need different — sometimes conflicting — versions of the same library. Install everything into one global Python and you eventually hit "dependency hell", where upgrading a package for one project breaks another. The fix is to give each project its own environment: an isolated set of packages that can't interfere with anything else, and that you can describe in a file precisely enough to recreate months later.

For scientific work the group uses conda (or its faster drop-in mamba), because it installs not just Python packages but the compiled libraries scientific tools depend on. This module covers environments on your own machine; the cluster provides software a related but distinct way, which §6 hands off to Module 06 — HPC & SLURM Job Submission.

1 · Why environments

  • Isolation. Project A can use one version of a library and Project B another, with no collision.
  • Reproducibility. A recorded environment means the software behind a result is known exactly — essential when a calculation must be defended or repeated.
  • Cleanliness. Delete an environment and everything it pulled in is gone; your system Python stays untouched.

2 · conda and mamba

Install a minimal conda distribution — Miniforge is the group's recommendation (it defaults to the community conda-forge channel and ships mamba). Then create and use an environment:

conda create -n md numpy matplotlib mdanalysis   # new env named "md"
conda activate md                                 # switch into it
conda install -c conda-forge scipy                # add a package (conda-forge channel)
conda list                                        # what's installed here
conda deactivate                                  # leave the env

mamba is a faster, drop-in replacement for the conda command — mamba create, mamba install — useful when conda's dependency solver is slow.

3 · Reproducible environments with environment.yml

An environment is only reproducible if it's written down. Capture one, commit the file, and anyone (including future-you) can recreate it exactly:

conda env export --from-history > environment.yml   # a clean, portable recipe
conda env create -f environment.yml                 # recreate it elsewhere

Keep environment.yml in the project's Git repository (Module 03). --from-history records only the packages you actually asked for, which stays portable across operating systems.

4 · pip vs conda

Both install packages; the rule of thumb: prefer conda for anything with compiled/native dependencies (numpy, scipy, the MD tools), and use pip for pure-Python packages that aren't on conda-forge. If you must mix them, install the conda packages first, then pip into the activated environment — never system-wide.

5 · Installing the group's scientific stack

Most of the group's tools are on conda-forge; a couple are installed differently. Always check each tool's official docs for the current instructions.

  • MDAnalysis, numpy, matplotlib, scipy — conda install -c conda-forge mdanalysis (pulls in the rest). mdanalysis.org
  • LAMMPS and GROMACS — available on conda-forge for small/local test runs (conda install -c conda-forge lammps / gromacs); real runs use the cluster. LAMMPS · GROMACS
  • Packmol — on conda-forge, or build from its releases. m3g.github.io/packmol
  • VMD — a graphical application: download and install it directly for local visualisation. ks.uiuc.edu/Research/vmd

Local installs are for building and testing

Installing the MD engines locally is handy for building a system and debugging a tiny run. Production-scale simulations belong on the cluster — see Module 09 — Molecular Simulations §"Software you need" for how the two fit together.

6 · Local environments vs the cluster

Everything above is about software on your own machine. The HPC cluster provides software a different way — through environment modules and ready-made conda environments — which is taught in Module 06 — HPC & SLURM Job Submission §4. The short version of the hand-off:

  • On your machine: you create and own conda environments, as in this module.
  • On the cluster: you load software the platform already provides rather than installing it yourself; Module 06 covers exactly how.

Both worlds meet in the scientific modules — Module 09 — Molecular Simulations §"Software you need" walks through choosing local versus cluster software for an actual simulation.

Exercise / Deliverable

  1. Install Miniforge, then create a conda environment containing numpy, matplotlib, and mdanalysis.
  2. Activate it and confirm the packages import (python -c "import numpy, matplotlib, MDAnalysis").
  3. Export it to environment.yml with --from-history, then delete the environment and recreate it from the file.
  4. Run the Module 04 plotting exercise inside this environment to confirm it works end-to-end.

Deliverable: your environment.yml and a short transcript showing the environment created, exported, recreated, and the packages importing cleanly.
