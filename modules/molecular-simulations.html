Molecular Simulations

Molecular simulation is the group's core method. The hard part is not pressing "run" — it is building a sensible system, knowing when it has equilibrated, and trusting the numbers you get out. This module is built around good practice, not button-pushing.

Learning objectives

  • Build a starting configuration with Packmol
  • Understand MD integration, ensembles, thermostats and force fields
  • Follow the generation → equilibration → production workflow
  • Decide when a system is actually equilibrated
  • Analyse trajectories (RDF, MSD, RMSD)

Prerequisites

  • Module 07 — Statistical Mechanics
  • Module 03 — HPC & SLURM (to run jobs)
  • Basic Python — reading files, numpy arrays, matplotlib plots (no dedicated module yet)
  • Assumes Module 01 — Unix & Module 02 — Bash (SSH, running scripts)

Estimated time

  • 45–50 hours

Overview

Molecular dynamics (MD) integrates Newton's equations for every atom to generate a trajectory; Monte Carlo (MC) samples configurations stochastically. Both are machines for drawing samples from the ensembles you met in Module 07 — so that a time or ensemble average over the trajectory estimates a thermodynamic property. Everything below is in service of one goal: making those averages trustworthy.

Software you need

This module uses Packmol (system building), an MD engine — the group mainly uses LAMMPS and GROMACS — plus MDAnalysis (Python trajectory analysis) and VMD (visualisation). Where each lives depends on whether you are testing or doing real work.

  • On the cluster (all real runs). Don't install the engines yourself — they are provided as environment modules. Look up the name on the cluster and load exactly what you find; the exact module strings are not fixed, so confirm them with module avail rather than trusting a name written here: 
    module avail                     # everything installed
module avail | grep -i lammps    # find the engine you want
module load <name-you-found>      # the exact string module avail printed
    Python tools such as MDAnalysis come from a Conda environment: after module load anaconda3, use conda env list and source activate <env> exactly as in Module 03 §4. Install packages into your own environment only inside an interactive session or a job — never build or install heavy software on the shared login node.
  • On your own machine (tiny test systems only). A local install is handy for building and debugging a small box before you submit. Install from each tool's official docs — Packmol, LAMMPS, GROMACS, MDAnalysis — and keep systems small. Production-scale work always goes to the cluster.
  • VMD is a graphical application: run it locally to inspect trajectories, not on a login node.

1 · Building the system (generation)

A simulation is only as good as its starting point. You need atoms placed without unphysical overlaps, at roughly the right density, in a box with the right boundary conditions. Packmol packs molecules into a region while respecting minimum-distance constraints — ideal for building solvent boxes, mixtures, and polymer melts. You then assign a force field (OPLS, CHARMM, AMBER, …) that defines the bonded and non-bonded interactions.

Get the density and the force field right first

A box that is too dense will explode on the first step; the wrong force field gives precise answers to the wrong question. Sanity-check both before you run anything long.

2 · The MD engine

MD advances the system in small timesteps (typically ~1–2 fs) with a symplectic integrator — velocity Verlet. You choose an ensemble to match your question (see Module 07): NVE for pure dynamics, NVT to control temperature (via a thermostat — Nosé–Hoover, Langevin), NPT to control pressure too (via a barostat). Periodic boundary conditions remove surface effects so a small box mimics bulk. The group mainly uses LAMMPS and GROMACS.

Monte Carlo takes a different route: propose a move, accept it with the Metropolis probability min(1, e^(−ΔE/k_BT)), repeat. MC is powerful when you care about equilibrium configurations rather than real-time dynamics.

3 · The good-practice workflow

Never run a production simulation straight from a freshly built box. The disciplined pipeline is generation → equilibration → production, and the middle stage is where most mistakes happen.

How do you know it's equilibrated? Don't guess from the clock — check that the relevant quantities have stopped drifting:

  • Energy (potential/total) has plateaued — no systematic upward or downward trend.
  • Temperature fluctuates around the target, not toward it.
  • Density / box volume (in NPT) has levelled off.
  • Structural measures (e.g. the RDF, radius of gyration for polymers) are stable between the first and second half of the window.
  • Use block averaging to estimate uncertainties and confirm the running average has converged. Discard the pre-equilibrium portion before averaging.

Make these checks operational rather than eyeballed:

  • Fit the drift. Fit a straight line to the potential (or total) energy over the last ~20% of the run and treat it as equilibrated when the slope is statistically consistent with zero. A useful rough target is a slope well under a fraction of a percent of the mean energy per nanosecond — but calibrate the threshold to your own system rather than treating any single number as universal.
  • Split-half test. Compute a structural quantity (e.g. the RDF) or the density separately from the first and second halves of a window; if the two overlap within noise, that window is stationary — if they differ visibly, keep equilibrating.
  • Block averaging, concretely. Divide the post-equilibration data into blocks of increasing size; as the block size grows, the estimated standard error of the mean stops shrinking once the blocks are effectively independent. That plateau is both your convergence signal and an honest error bar.

4 · Analysing trajectories

The production trajectory is your data. The workhorses:

  • Radial distribution function g(r) — local structure: how likely you are to find a particle at distance r from another.
  • Mean-squared displacement (MSD) — from its slope you get the diffusion coefficient (Einstein relation).
  • RMSD — deviation from a reference structure over time (common for biomolecules).

Compute these with tools like MDAnalysis (Python) or the engine's own analysis utilities, and visualise trajectories with VMD.

What "Basic Python" means here: reading a trajectory with MDAnalysis, computing simple quantities with numpy, and plotting them with matplotlib. If those are new to you, work through a short numpy + matplotlib tutorial before the exercise — there is no dedicated Python module in the curriculum yet, so ask the group for a current recommendation.

5 · Running on the cluster

Real simulations run on the HPC cluster, not your laptop. This section shows the shape of a simulation job; Module 03 §5–§6 is the full SLURM reference — commands, partitions, monitoring, etiquette — and everything here builds on it rather than repeating it.

Find and load the engine

Engines are environment modules — don't guess the name, look it up on the cluster and load exactly what you find, then confirm it starts:

module avail | grep -i lammps    # or gromacs, packmol, ...
module load <name-you-found>      # the exact string module avail printed
lmp -help    #  or:  gmx --version     # confirm the engine actually runs

Every specific below is an example — verify it on the cluster

The module names, wall-times, and core / GPU counts in the scripts that follow are placeholders to right-size for your own system, not established cluster facts. Confirm module names with module avail, and set the resource requests from a short test run. Only the partition and QoS names (cpu6348 / qos=26cores and gpua800 / qos=4a800) are fixed — they are the ones documented in Module 03 §5.

A CPU job (e.g. LAMMPS)

#!/bin/bash
#SBATCH --job-name=md-prod
#SBATCH --partition=cpu6348       # CPU partition (Module 03)
#SBATCH --qos=26cores             # QoS matching that partition
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=26      # cores → MPI ranks; right-size for your system
#SBATCH --time=24:00:00           # wall-time estimate — verify from a short run
#SBATCH --output=%j.out
#SBATCH --error=%j.err

module load <lammps-module>        # find the exact name: module avail | grep -i lammps
srun lmp -in in.production         # srun launches one rank per --ntasks

A GPU job (e.g. GROMACS)

#!/bin/bash
#SBATCH --job-name=md-prod-gpu
#SBATCH --partition=gpua800        # GPU partition (Module 03)
#SBATCH --qos=4a800                # QoS matching that partition
#SBATCH --gres=gpu:1               # number of GPUs — start with 1
#SBATCH --cpus-per-task=8          # CPU cores feeding the GPU; right-size
#SBATCH --time=24:00:00            # wall-time estimate — verify from a short run
#SBATCH --output=%j.out
#SBATCH --error=%j.err

module load <gromacs-module>        # find the exact name: module avail | grep -i gromacs
gmx mdrun -deffnm prod -nb gpu     # run the pre-built prod.tpr on the GPU

Submit, watch, and collect exactly as in Module 03 §6: sbatch prod.slurm, then squeue -u $USER, then read <jobid>.out.

Before you submit

  • Your input and topology files are on the shared cluster file system, not just your laptop.
  • You have run the same system for a handful of steps locally or in a short interactive session — it starts and doesn't blow up.
  • You have estimated wall-time from that short run (ns/day) and requested a little more — not a blind 7 days.
  • You are requesting only the cores / GPUs the engine will actually use; idle reserved resources block everyone else (Module 03 §7).

6 · Common mistakes

Catch these before they cost you a run

  • Skipping energy minimisation. Starting MD from an un-minimised box almost always blows up in the first few steps.
  • Starting production too early. If the density is still drifting in NPT the system hasn't equilibrated, and the data is contaminated by the starting configuration.
  • Too large a timestep. Use ~1 fs for all-atom force fields; 2 fs only when the fastest bonds are constrained (SHAKE / LINCS). A too-large step silently corrupts the dynamics or crashes the run.
  • Production in NVE by accident. Without a thermostat the temperature drifts — use NVT/NPT unless you specifically want conserved-energy dynamics.
  • Output frequency too coarse. If the thermo / log output is too infrequent you cannot see whether equilibration happened at all.
  • Never testing small first. Debug on a tiny system for a few steps before committing a long, many-core job.
  • Trusting one short run. A single trajectory has no error bar — plan for block averaging or replicate runs (see §3).

Exercise / Deliverable

  1. Set up your tools. Get Packmol, an MD engine, and MDAnalysis working — locally for building, on the cluster for running — and verify each one starts with a trivial command (e.g. lmp -help, gmx --version).
  2. Build a simple system with Packmol (e.g. a Lennard-Jones fluid or a small solvent box).
  3. Minimise, then equilibrate in NVT and NPT; plot energy, temperature and density vs. time and mark the point at which each has converged, justifying it with a drift or split-half check (§3).
  4. Run a short production simulation on the cluster with sbatch. Record the job ID, the squeue -u $USER line showing it running, and the wall-time it actually used versus what you requested.
  5. From the production trajectory, compute and plot the RDF and the MSD (and estimate the diffusion coefficient from the MSD slope).
  6. In one or two sentences, name one thing you would change on a re-run (longer equilibration, a different timestep, more replicate runs, …) and why.

Deliverable: your input/build files, the equilibration plots with the equilibration point marked and justified, the RDF/MSD figures, your job script plus the squeue/wall-time evidence from step 4, and your one-line reflection from step 6.
