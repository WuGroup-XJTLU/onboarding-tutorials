Bash Scripting

A script turns a sequence of commands you keep re-typing into one reproducible, shareable file. This module covers the language features you need to write scripts that are correct, robust, and readable.

Learning objectives

  • Write and run a Bash script from a shebang
  • Use variables, quoting, and command substitution correctly
  • Take command-line arguments
  • Branch and loop over files
  • Factor code into functions and fail loudly on errors

Prerequisites

  • Module 01 — Unix/Linux Basics

Estimated time

  • 15–18 hours

Overview

Any time you find yourself running the same three commands over and over, or copy-pasting a sequence from your notes, that sequence should become a script. A script is a plain text file of shell commands that anyone (including future-you) can read, re-run, and trust to do the same thing every time. Reproducibility is the whole point: a result that came from a script you kept is a result you can defend and repeat. Keep those scripts under version control (Module 03 — Version Control with Git) so their full history is recoverable too.

This module is about writing scripts well. It does not cover scheduling/automation (cron and friends) — for running scripts as cluster jobs, see Module 06.

The anatomy of a script

A script starts with a shebang telling the system which interpreter to use, and is made executable with chmod +x.

Variables & quoting

Assign with name=value (no spaces around =) and expand with $name. Always quote your expansions — "$var" — so that values containing spaces don't get split into multiple words. This single habit prevents most beginner bugs.

greeting="hello world"
echo $greeting        # two words after splitting — usually not what you want
echo "$greeting"      # one argument, exactly as stored  ✓

today=$(date +%F)     # command substitution: capture a command's output
echo "Log for ${today}"

Arguments

Values passed on the command line appear as $1, $2, …; $@ is all of them and $# is the count.

#!/usr/bin/env bash
# usage: ./analyse.sh INPUT_DIR [OUTPUT_DIR]
indir="$1"
outdir="${2:-results}"      # default to "results" if $2 is unset
echo "reading from $indir, writing to $outdir"

Conditionals

Use [[ ... ]] for tests. Common ones: -f (file exists), -d (directory exists), -z (empty string), string ==, and numeric -lt -le -gt -ge -eq.

if [[ ! -d "$indir" ]]; then
  echo "error: $indir is not a directory" >&2   # message to stderr
  exit 1                                          # non-zero = failure
fi

if [[ "$#" -lt 1 ]]; then
  echo "usage: $0 INPUT_DIR [OUTPUT_DIR]" >&2
  exit 2
fi

Loops

# iterate over files safely (globs, not `ls`)
for f in "$indir"/*.xyz; do
  echo "processing $f"
  # ... do work on "$f"
done

# repeat until a condition changes
n=0
while [[ "$n" -lt 5 ]]; do
  echo "step $n"
  n=$(( n + 1 ))           # arithmetic
done

Functions

log() {                     # define a reusable helper
  echo "[$(date +%T)] $*"
}

convert() {
  local src="$1"            # `local` keeps variables scoped to the function
  local dst="${src%.xyz}.pdb"
  some_tool "$src" > "$dst"
  log "wrote $dst"
}

log "starting"
convert input.xyz

Failing loudly: set -euo pipefail

By default Bash keeps going after a command fails, which can silently corrupt a run. Put this near the top of every script:

set -euo pipefail
# -e  exit immediately if any command fails
# -u  error on use of an unset variable (catches typos)
# -o pipefail  a pipeline fails if ANY stage fails, not just the last

Lint your scripts

Run ShellCheck on every script — it catches quoting mistakes, unset variables, and dozens of other footguns automatically. Many editors run it as you type.

Resources

  • Bash Guide for Beginners — Machtelt Garrels (TLDP). A gentle, complete introduction. tldp.org/LDP/Bash-Beginners-Guide
  • Advanced Bash-Scripting Guide — Mendel Cooper (TLDP). A deep reference for when you need more. tldp.org/LDP/abs
  • ShellCheck — a static analysis tool that finds bugs in shell scripts. shellcheck.net
  • Google Shell Style Guide — sensible conventions used across many labs. google.github.io/styleguide/shellguide

Exercise / Deliverable

Write a script summarise.sh that:

  1. Takes an input directory as $1 and prints a usage message and exits non-zero if it is missing or not a directory.
  2. Starts with set -euo pipefail and a shebang.
  3. Loops over every *.dat file in the directory and, for each, prints the filename and its line count.
  4. Uses a function to do the per-file work.
  5. Passes ShellCheck with no warnings, and exits 0 on success.

Deliverable: the summarise.sh file plus a screenshot or transcript of it running on a test directory and its clean ShellCheck output.
