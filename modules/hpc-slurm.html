HPC & SLURM Job Submission

The group's real computing runs on the XJTLU High Performance Computing (HPC) Platform. This module takes you from "I have an account" to "my job is running on a compute node": connecting, loading software, writing a SLURM batch script, and cluster etiquette.

Learning objectives

  • Get and keep an HPC account
  • Connect and transfer data
  • Load software with environment modules
  • Write, submit, and monitor a SLURM job
  • Follow cluster etiquette

Prerequisites

  • Module 01 — Unix/Linux Basics
  • Module 02 — Bash Scripting

Estimated time

  • 15–20 hours

Overview

An HPC cluster is not one big computer — it is many compute nodes you reach through a shared login node, all seeing the same file system, with a scheduler (SLURM) deciding whose job runs where and when. You do not run heavy work directly; you describe it in a small script and submit it, and the scheduler runs it on a compute node when resources are free.

This module is written for the XJTLU platform specifically. The authoritative, always-current reference is the XJTLU HPC documentation (reachable on the campus network / via the university VPN) — check it for the latest node lists and limits.

1 · Get an account

You need a user account and membership of a project that has an allocation. Apply through the HPC Request Application Form on the XJTLU intranet; as a student you apply with your supervisor's (project leader's) approval. After you receive the account, change the password within three days (log in and run passwd); accounts idle for six months may be suspended, and a forgotten password is reset through Esupport.

2 · The cluster at a glance

The platform spans two campuses (SIP and XEC) and is made of login nodes, data-transfer nodes, compute nodes (CPU, huge-memory, and GPU), and a shared GPFS file system, all linked by a fast InfiniBand HDR100 network.

3 · Connect & transfer data

Log in over SSH to the login node for your campus (from off-campus you must be on the XJTLU VPN first):

ssh <username>@login.hpc.xjtlu.edu.cn      # SIP campus
ssh <username>@xeclogin.hpc.xjtlu.edu.cn   # XEC campus

On Windows, MobaXterm or PuTTY are the recommended clients. Move files through the data-transfer node (dtn.hpc.xjtlu.edu.cn, or xecdtn.hpc.xjtlu.edu.cn) with scp/rsync (or WinSCP on Windows):

scp -r ./inputs  <username>@dtn.hpc.xjtlu.edu.cn:~/runs/
rsync --archive --partial --progress ./runs/  <username>@dtn.hpc.xjtlu.edu.cn:~/runs/

Two conveniences worth setting up once on your own machine in ~/.ssh/config: a short Host alias so you can type ssh hpc instead of the full address, and a persistent (multiplexed) connection so repeated logins reuse one authenticated channel rather than prompting every time:

Host hpc
    HostName login.hpc.xjtlu.edu.cn
    User <username>
    ControlMaster auto                 # reuse one connection...
    ControlPath  ~/.ssh/cm-%r@%h:%p    # ...over this socket...
    ControlPersist 10m                 # ...kept alive for 10 min

After this, ssh hpc (and scp file hpc:~/runs/) just work, and a second session opens instantly without re-authenticating.

4 · Load software with modules

Software is provided as environment modules — you load what you need rather than installing it yourself:

module avail                 # list available software
module load anaconda3        # e.g. load Anaconda (Python)
module list                  # what's loaded now
module purge                 # unload everything

Compilers and libraries (gcc, Intel oneAPI/MKL/MPI, OpenMPI, CMake, CUDA) load the same way. For Python, the platform ships ready-made Conda environments — after module load anaconda3, run conda env list to see them (e.g. base, pytorch-base, tf-env-base), and source activate <env> to use one.

5 · SLURM: the scheduler

The golden rule

The login node is shared by everyone — never run heavy or parallel work on it. Put your work in a batch script and submit it so SLURM runs it on a compute node.

The commands you will use most:

sinfo         # cluster / partition status
squeue        # the job queue (add -u $USER for just yours)
sbatch job.slurm   # submit a batch job
scancel <jobid>     # cancel a job
sacct         # reports on your completed jobs
slurm-tool    # your storage quota, Account, QoS and limits at a glance

slurm-tool is an XJTLU-specific convenience wrapper, not part of standard SLURM — don't expect to find it on other clusters. The rest (sinfo, squeue, sbatch, scancel, sacct) are standard SLURM and carry over anywhere.

Jobs run in partitions (queues), each paired with a QoS that sets limits (the maximum wall-time is up to 7 days):

cpu6348

  • general CPU jobs
  • 1–26 cores by default

cpuhuge1t

  • huge-memory jobs
  • 1–92 cores by default

gpu3090

  • 1–8 × NVIDIA RTX 3090

gpua800

  • 1–4 × NVIDIA A800

Use sinfo for the live list (partitions and limits change — the HPC docs are authoritative). salloc exists for quick interactive debugging only — release it as soon as you're done, and use sbatch for real work.

6 · Write & submit a job script

A SLURM batch script is a normal Bash script whose top comment lines (#SBATCH) request resources. A CPU example:

#!/bin/bash
#SBATCH --job-name=hello         # a name shown in the queue
#SBATCH --partition=cpu6348      # which partition/queue
#SBATCH --qos=26cores            # QoS matching the partition's limits
#SBATCH --nodes=1                # number of nodes
#SBATCH --ntasks-per-node=4      # cores on that node
#SBATCH --output=%j.out          # stdout  → <jobid>.out
#SBATCH --error=%j.err           # stderr  → <jobid>.err

module load anaconda3            # load the software you need
hostname                         # which compute node am I on?
python my_analysis.py            # ... the actual work

Submit it, watch it, and collect the output:

$ sbatch hello.slurm
Submitted batch job 2023
$ squeue -u $USER            # PENDING → RUNNING → (gone when done)
$ ls 2023.out 2023.err       # results appear in the files you named

A GPU job is the same with a GPU partition and a --gres request:

#SBATCH --partition=gpua800
#SBATCH --qos=4a800
#SBATCH --gres=gpu:2             # request 2 GPUs
#SBATCH --cpus-per-task=4

7 · Etiquette

  • Never compute on the login node — it's shared; submit a job instead.
  • Request only what you'll use. Idle reserved cores/GPUs block other people; the cluster is a shared, scarce resource.
  • Release interactive sessions (salloc) as soon as you finish debugging.
  • Name files and directories with plain English characters and numbers — special characters can break jobs.
  • Mind your storage quota (check with slurm-tool); write large job output to your allocated space, not $HOME forever.

Resources

  • XJTLU HPC documentation — the authoritative reference (accounts, login, software, SLURM, examples). On campus / via VPN. hpc.xjtlu.edu.cn
  • SLURM official documentation — the scheduler's own manuals for sbatch, srun, squeue, etc. slurm.schedmd.com
  • Module 01 — Unix/Linux Basics and Module 02 — Bash Scripting — SSH, the shell, and scripting all carry directly over to the cluster.

Exercise / Deliverable

  1. Log in to the cluster and confirm you're on a login node; run slurm-tool to see your Account, QoS, and storage quota.
  2. Run sinfo and note which partitions are available and idle.
  3. Write a small hello.slurm that loads a module, prints the compute node's hostname, and runs one trivial command; submit it with sbatch.
  4. Watch it with squeue -u $USER, then retrieve and show your <jobid>.out file.

Deliverable: your hello.slurm script and the contents of its output file, plus one sentence naming the compute node it ran on.
