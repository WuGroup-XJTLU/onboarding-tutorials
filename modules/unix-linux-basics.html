Unix/Linux Basics

Almost all of the group's computing happens on Linux — your laptop's terminal, the group workstations, and the XJTLU cluster. This module gets you fluent in the shell so everything after it feels natural.

Learning objectives

  • Move around the filesystem and manage files from the command line
  • Read and set file permissions
  • Combine small tools with pipes and redirection
  • Inspect and control processes
  • Log in to a remote machine and copy files to it

Prerequisites

  • A terminal (macOS/Linux built-in; Windows: WSL or MobaXterm)
  • No prior command-line experience assumed

Estimated time

  • 12–15 hours

Overview

A shell is a program that reads the commands you type and asks the operating system to run them. On Linux and macOS the usual shell is bash (or zsh, which behaves almost identically for everything here). Working in the shell feels slow for a day and then becomes far faster than clicking: it is scriptable, it works identically on your laptop and on a supercomputer, and it is how you will drive every simulation and analysis tool the group uses.

You do not need to memorise everything. Aim to recognise what is possible and to know how to look things up — man <command> and <command> --help are always there, and explainshell.com will annotate any command you paste into it.

The filesystem & moving around

Unix organises everything into one tree that starts at the root, /. Your files live under your home directory, written ~ (e.g. /home/alice). A path is an address in this tree: absolute paths start at /, relative paths start from where you currently are.

The core navigation commands:

pwd                 # print working directory — "where am I?"
ls                  # list files here
ls -la              # long listing, including hidden (dot) files
cd projects         # change into ./projects (relative)
cd /home/alice      # jump somewhere by absolute path
cd ..               # go up one level
cd                  # go home (~)

Working with files & directories

mkdir -p runs/2026/test    # make directories, parents included
cp  a.dat backup/          # copy
mv  old.dat new.dat        # move / rename
rm  junk.log               # remove a file  (there is no undo — be careful)
rm -r olddir/              # remove a directory and its contents
cat  small.txt             # dump a file to the screen
less big.log               # scroll a large file (q to quit, / to search)
head -n 20 f.csv           # first 20 lines;  tail -n 20  for the last 20
find . -name "*.xyz"       # search the tree for matching files

On deleting things

rm is permanent — there is no recycle bin. Double-check the path before you run rm -r, and never run rm -rf / or a stray rm -rf * in the wrong directory. When unsure, ls first to see exactly what a wildcard matches.

Permissions

Every file has an owner, a group, and three sets of permissions — read, write, execute — for the owner, the group, and everyone else. ls -l shows them as a ten-character string like -rwxr-xr--.

chmod +x run.sh          # make a script executable
chmod 644 notes.txt      # rw-r--r--  (owner writes, others read)
chmod 755 bin/           # rwxr-xr-x  (typical for a directory)
chown alice:group f.dat  # change owner/group (needs privileges)

The three-digit form treats rwx as bits: r=4, w=2, x=1, summed per field. So 7 = rwx, 6 = rw-, 5 = r-x.

Pipes, redirection & text tools

The Unix philosophy is small tools that each do one thing, combined with pipes. A pipe | sends one command's output into the next command's input; redirection > sends output into a file.

grep "pattern" file        # print lines matching a pattern
wc -l file                 # count lines  (-w words, -c bytes)
sort file | uniq -c        # sort, then count unique lines
cut -d, -f2 data.csv       # take the 2nd comma-separated column
command > out.txt          # redirect output to a file (overwrite)
command >> out.txt         # append instead of overwrite
command 2> err.txt         # redirect errors (stream 2) separately

Processes

A running program is a process. You will need to watch long simulations and occasionally stop a runaway job.

top    (or htop)           # live view of running processes (q to quit)
ps aux | grep lammps       # find a specific process
./long_job.sh &            # run in the background; note the job/PID
jobs                       # list background jobs in this shell
kill  12345                # ask process 12345 to stop
kill -9 12345              # force-kill if it will not stop
Ctrl-C                     # interrupt the foreground command
Ctrl-Z then bg             # suspend, then resume in the background

Remote access

The group's real computing runs on remote machines. You reach them with ssh and copy files with scp or rsync. (Getting onto the XJTLU cluster specifically — VPN, login nodes, and the SLURM scheduler — is covered in Module 03.)

ssh alice@server.example.edu        # open a shell on a remote machine
scp results.dat alice@server:~/in/  # copy a local file to the remote
scp alice@server:~/out/traj.xtc .   # copy a remote file to here
rsync -av runs/ alice@server:~/runs/ # sync a directory (efficient, resumable)

Set up SSH keys early

Generate a key pair with ssh-keygen and copy the public key to the server (ssh-copy-id). You then log in without typing a password every time — and scripts can connect non-interactively.

A terminal editor

You will often edit files without a graphical editor — over SSH, for example. Learn at least one terminal editor. nano is the gentlest (shortcuts shown at the bottom of the screen; Ctrl-O saves, Ctrl-X exits). vim is more powerful and everywhere; if you open it by accident, press Esc then type :q! to quit without saving.

Resources

  • The Unix Workbench — Sean Kross. A free, beginner-friendly online book that mirrors this module closely. seankross.com/the-unix-workbench
  • Software Carpentry — The Unix Shell — a hands-on lesson widely used to teach researchers the shell. swcarpentry.github.io/shell-novice
  • explainshell — paste any command to see each flag explained. explainshell.com
  • man pages & --help — your first stop for any command, always available offline.

Exercise / Deliverable

From a fresh terminal, do the following and keep a record of the commands you used:

  1. Create the directory structure ~/onboarding/unix/{data,scripts,results} in a single command.
  2. Download or create a text file of at least 100 lines in data/. Using a pipeline, count how many lines contain a word of your choice and save that number to results/count.txt.
  3. Write a one-line shell script scripts/hello.sh that prints a message, make it executable with chmod, and run it.
  4. Show the permissions of everything in scripts/ with ls -l and explain, in a sentence, what the permission string on hello.sh means.
  5. If you have access to a remote machine (or the cluster), scp your results/count.txt to it and back.

Deliverable: a short text file (or notebook) listing the commands you ran for each step and their output.
